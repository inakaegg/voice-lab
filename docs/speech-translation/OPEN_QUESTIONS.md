# 未確定事項

## 決定済み

1. 声質クローン
   - 最終的には必須。
   - アプリの売りは、入力話者の声と出力音声の声が同じになること。
   - 初期実装では声質クローンなしでよい。まず `音声 -> 文字起こし -> 翻訳 -> 音声` の縦切りを通す。

2. 中国語出力
   - `zh-CN` を既定にする。
   - ここでの `zh-CN` は普通話を前提にした中国大陸向けの簡体字中国語を指す。

3. 末尾付加のテキスト加工
   - MVP範囲に含める。
   - ただし、最初の実装では後回しでよい。
   - 固定の `モー` 加工ではなく、指定した文字列を末尾に付加する機能として扱う。
   - 既定の翻訳ルールではなく、任意の `text_transform=append_suffix` として扱う。

4. 応答速度
   - できるだけ速く出力する。
   - 理想は、スマホに向かって話し終えた直後に出力音声が話し始めること。
   - 最初のAPIは同期形式でよいが、将来のストリーミング出力や非同期job形式を妨げない構造にする。

5. 外部有料API
   - 完全除外はしない。
   - RunPod等で自前運用するコストや手間と比較して検討する。
   - 導入する場合は、目的、費用、依存リスク、キー管理を実装前にdocsへ明記する。

6. モデルキャッシュ
   - モデルキャッシュはリポジトリ外に置く。

## まだ決めること

1. 入力音声を保存するか。
   - 現在の草案では、既定で保存しない。
   - voice profileを保存する場合は、別仕様が必要。

2. 最初のデプロイ先をRunPodにするか、Modalも先に比較するか。
   - 現在の草案では、まずローカルMVPを作る。
   - その後、選んだPythonモデル構成でModalの方が明らかに簡単でない限り、RunPod Serverless + Network Volumeを第一候補にする。

## 早めに決めたいこと

1. ローカル開発時のキャッシュ置き場。
   - リポジトリ外の明示的な `MODEL_CACHE_DIR` を使う。
   - 実装時に環境変数名、既定値、未設定時のエラー表示を固定する。

2. 初期ローカルTTSプロバイダ。
   - macOSのOS TTSは最短で組み込みやすいが、Linuxデプロイにはそのまま移植できない。
   - GPU TTSプロバイダは、同じinterfaceの裏側に後から追加する。

3. サーバGPU版で最初に試すモデル構成。
   - 最小構成: ASR + 翻訳 + Qwen3-TTSまたはCosyVoice。
   - 声の類似度を重視する構成: まず直接声質クローンTTSを評価し、それで不足する場合にSeed-VCを追加する。

4. 末尾付加の単位。
   - 文ごとに付加するか、出力全体の末尾に1回だけ付加するかを決める。
   - 例: `こんにちはX。ありがとうX。` と `こんにちは。ありがとう。X` のどちらを採用するか。

5. UIの対象。
   - 最初はデスクトップブラウザだけにするか、スマホブラウザ録音も最初から含めるか。
   - スマホブラウザ対応を含める場合、録音形式と権限まわりの確認が増える。

6. デプロイ自動化の段階。
   - 最小: プラットフォームCLI/APIを呼ぶスクリプト。
   - 次段階: GitHub Actionsでimage buildとendpoint更新。
   - Terraformは、プラットフォーム選定が固まってから検討する。

# 未確定事項

## 決定済み

1. 声質クローン
   - 最終的には必須。
   - アプリの売りは、入力話者の声と出力音声の声が同じになること。
   - 初期実装では声質クローンなしでよい。まず `音声 -> 文字起こし -> 翻訳 -> 音声` の縦切りを通す。

2. 中国語出力
   - `zh-CN` を既定にする。
   - ここでの `zh-CN` は普通話を前提にした中国大陸向けの簡体字中国語を指す。

3. 末尾付加のテキスト加工
   - MVP範囲に含める。
   - ただし、最初の実装では後回しでよい。
   - 固定の `モー` 加工ではなく、指定した文字列を末尾に付加する機能として扱う。
   - 既定の翻訳ルールではなく、任意の `text_transform=append_suffix` として扱う。

4. 応答速度
   - できるだけ速く出力する。
   - 理想は、スマホに向かって話し終えた直後に出力音声が話し始めること。
   - 最初のAPIは同期形式でよいが、将来のストリーミング出力や非同期job形式を妨げない構造にする。

5. 外部有料API
   - 完全除外はしない。
   - RunPod等で自前運用するコストや手間と比較して検討する。
   - 導入する場合は、目的、費用、依存リスク、キー管理を実装前にdocsへ明記する。

6. モデルキャッシュ
   - モデルキャッシュはリポジトリ外に置く。

7. ローカル実プロバイダ
   - `MO_PROVIDER_MODE=local` では、faster-whisper、Qwen3翻訳、Qwen3-TTS/Seed-VCを既定で使う。
   - OpenAI Whisper smallとNLLB distilled 600Mは品質要件を満たさなかったため、実装候補から外す。
   - 詳細は `docs/speech-translation/LOCAL_PROVIDERS.md` を正とする。

## まだ決めること

1. 声質クローン本実装で最初に評価するモデル。
   - 候補は `docs/speech-translation/VOICE_CLONE.md` を正とする。
   - 追加依存の導入、GPU環境での評価、または外部API比較のいずれかを決める必要がある。

2. 入力音声を保存するか。
   - 現在の草案では、既定で保存しない。
   - voice profileを保存する場合は、別仕様が必要。

3. 最初のデプロイ先をRunPodにするか、Modalも先に比較するか。
   - 現在の草案では、まずローカルMVPを作る。
   - その後、選んだPythonモデル構成でModalの方が明らかに簡単でない限り、RunPod Serverless + Network Volumeを第一候補にする。

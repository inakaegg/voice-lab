# SpeakLoop ロードマップ

更新日: 2026-07-20

## この文書の役割

この文書は、SpeakLoopを今後どの順序で改善するかを示す。
現在実装されている挙動は [SPEC.md](SPEC.md) を正とし、
現在の作業順や引継ぎは非公開の内部作業メモで管理する。

## 製品の方向

SpeakLoopは、利用者が母語で話した「言いたいこと」を学習対象言語へ変換し、
お手本を聞き、復唱し、言えるようになるまで繰り返す発音練習アプリとする。

現在の学習対象は、日本語話者が英語または中国語を練習する用途である。
翻訳機能そのものではなく、次の流れを1つの画面で繰り返せることを製品価値とする。

1. 母語で言いたいことを録音する。
2. 学習対象言語のお手本文と音声を作る。
3. お手本を聞いて復唱する。
4. 聞き取られた内容と改善点を確認する。
5. お手本と復唱をフレーズごとに聞き比べ、再挑戦する。

## 現在地

母語録音、お手本文と音声の生成、復唱録音、音声認識、採点、
フレーズごとの比較再生まで実装済みである。

フレーズ対応付けと採点は大規模言語モデル（LLM）による比較・採点のみを使う。
文字列類似度による旧採点・フレーズ対応付け処理は、ローカル版・Cloudflare版とも
削除済みである（詳細は次節）。

## LLMによる比較再生と採点

### LLMへ任せる処理

お手本と復唱の音声認識結果をLLMへ渡し、次を一度に行う。

- 目標文のフレーズ分割
- お手本と復唱の認識結果との対応付け
- 全体の採点とコメント
- フレーズごとの採点とコメント

LLMには、目標文、両方の音声認識本文、認識単位ごとの開始・終了時刻、
音声の長さ、前後共通の余白を渡す。LLMは、フレーズごとの対応状態
（`assigned`/`partial`/`missing`）、対応する認識単位の位置番号
（`word_start_index`/`word_end_index`）、点数、コメントを含む完成JSONを返す。

アプリは、フレーズを連結すると元の目標文になること、位置番号が入力の
認識単位配列の範囲内であることを検査する。値が不正ならエラーとし、
アプリ側でフレーズ分割、点数、位置番号の対応付けそのものは作り直さない。

一致文字列（`matched_text`）と、元の時刻・余白適用後の再生時刻は、
検査済みの位置番号が指す認識単位からアプリが直接計算する。これらは
位置番号が決まれば一意に定まる値であり、LLMに転記・再計算させる理由が
ないためである。

#### 設計変更の経緯（2026-07-19）

初期実装は、LLMが返す元の時刻・余白適用後の再生時刻・一致文字列を、
アプリが独自計算した期待値と完全一致（浮動小数点許容誤差1e-6）で検査
していた。位置番号の選択自体は正しくても、そこから導ける時刻をLLMが
転記・再計算する際に誤ることがあり、その場合は比較結果全体が拒否され
「比較結果を作成できませんでした。もう一度お試しください。」が表示
された。位置番号が決まればこれらの値は一意に計算できるため、LLMには
位置番号の選択という意味判断だけを任せ、時刻と一致文字列の計算はアプリ
側の固定式に一本化した。

### ユーザー画面

利用者が使う`/speakloop`画面に次を追加する。

- 比較モデル:
  `gpt-5.6-terra`を初期値とし、`gpt-5.6-luna`、`gpt-5.4-mini`、
  `gpt-5.4-nano`から選択できるようにする。
- 前後共通の再生余白:
  初期値`0.10`秒、`0.00`〜`0.50`秒、`0.05`秒刻みとする。

お手本と復唱の両方へ同じ余白を加える。
余白によって隣り合うフレーズの再生区間が重なってもよく、
アプリ側でLLMの対応付けを変更しない。

### ローカル診断ログ

ローカル版は、復唱1回につき1つのJSONログを次へ保存する。

```text
tmp/practice-llm-alignment/<timestamp>-<id>.json
```

ログには次を含める。

- 目標文、選択モデル、再生余白
- お手本と復唱の音声認識本文、認識単位ごとの時刻
- LLMへ送った内容、未加工の返却内容、JSONとして読み取った値
- フレーズごとの最終再生区間、点数、コメント
- API使用量、料金単価、推定料金、処理時間
- 失敗した処理段階と技術的なエラー内容

APIキー、認証情報、音声データそのものは保存しない。
Cloudflare公開版は、この完全なログを保存しない。

### 失敗時

API呼び出しの失敗、JSONの形式不正、認識単位の位置番号の不正が起きた場合は、
旧処理へ自動で切り替えず、比較再生と採点を中止して次を表示する。

```text
比較結果を作成できませんでした。もう一度お試しください。
```

### この変更の完了条件

- お手本と復唱の音声認識結果、LLM返却値、最終再生区間を1つのログで確認できる。
- 4モデルをユーザー画面から選択でき、実際のAPI呼び出しへ反映される。
- 前後共通の再生余白がお手本と復唱の全フレーズへ反映される。
- 点数とフレーズ分割をアプリ側で作り直さない。
- LLMによる処理が失敗した場合は旧処理へ切り替えない。
- ローカル版とCloudflare公開版が同じ形式の結果を返す。

### 現在の実装範囲

- ローカルFastAPI版は、LLM呼び出し、返却値検査、診断ログ、ユーザー画面の設定に対応した。
- Cloudflare Workerも同じ設計（LLMは位置番号だけを返し、時刻と一致文字列はアプリ側で計算する）でLLM比較・採点に対応した。英語は同期処理内、中国語はRunPod ASR完了後のpolling応答内でLLMを呼ぶ。比較モデルと前後共通余白は、中国語側はRunPod job完了までKV（`MO_SPEECH_KV`、ローカル開発時はメモリ）で引き継ぐ。
- Cloudflare版は診断ログ（`tmp/practice-llm-alignment/`相当）を持たない方針のため、API使用量・推定料金はユーザー画面が読まないフィールドとして応答に含めない。それ以外のユーザー画面が読むフィールド（`overall_score`、`overall_comment`、`llm_comparison`、`comparison_alignment`、`model_comparison_alignment`、`comparison_model`、`playback_padding_seconds`など）はローカル版と同じ形式で返す。
- ユーザー画面は、非同期jobの処理段階、サービス名、モデル名を継続取得して表示する。
- お手本音声ASRは、音声内容のハッシュ、言語、providerをキーにキャッシュする。
- 同じお手本音声は再ASRせず、毎回新しい復唱音声だけをASRする。
- ローカル中国語版では、キャッシュ済みのお手本ASRをRunPod jobと対応するローカル履歴へ保存する。この場合はお手本音声をRunPodへ送らず、FastAPI再起動後も保存したASRを使う。
- ローカル履歴が無効な場合は、お手本音声をRunPod jobへ含める。再起動後に必要情報を失うjobは作らない。
- Cloudflare版のお手本ASRキャッシュは`MO_SPEECH_KV`へ保存する。
- Cloudflare版のテストは`tests/practice_llm_worker.test.mjs`（`tests/test_practice_llm.py`と同じ実データfixtureを使う検証ロジックの単体テスト）と`tests/cloudflare_worker.test.mjs`（英語・中国語それぞれの成功/LLM検証失敗をmockしたHTTP経路テスト）で担保する。

### 旧比較処理の削除

現在はLLMによる比較・採点だけを使用する。次の旧比較処理は削除済みである。

1. Pythonに残っていた文字列類似度による採点、フレーズ対応付け、旧結果形式への変換処理
2. Cloudflare Workerに重複して残っていた同等の採点・フレーズ対応付け処理
3. `/api/practice/attempts` エンドポイントと `recording_intent: attempt` 分岐
4. 旧比較処理だけを検証していたテスト、fixture、評価スクリプト
5. 他文書から参照のない旧比較処理専用の公開文書

削除後は、お手本と復唱の音声認識結果をLLMへ送り、検査済みの返却値を
表示と比較再生へ渡す経路だけが残っている。LLM処理の失敗時に旧処理へ切り替える
分岐は設けていない。

## その後の変更: ローカル版と公開版の処理統一

現在FastAPIとCloudflare Workerへ重複して実装されている受付、外部サービス呼び出し、
結果の整形、エラー処理を、同じ実装へ統一する。

共通部分にはHonoを採用する。Honoは、Cloudflare Workersとローカルの
JavaScript実行環境で同じAPI処理を動かすためのWebフレームワークとして使用する。

1. Cloudflare Worker側のAPI処理をHonoへ移す。
2. ローカル版もHonoのAPI処理を入口として使用する。
3. Pythonが必要な音声処理やローカルモデル処理だけを補助APIとして残す。
4. HonoからPythonの補助APIを、ほかの外部サービスと同じ方法で呼び出す。
5. 同じ入力に対する結果形式、エラー、ログ項目の一致を確認する。
6. 確認後、FastAPIに重複して残る受付・制御処理を削除する。

FastAPIを直ちに全廃するのではなく、Python依存処理を提供する役割へ縮小する。

## 今後の変更: 公開文書の整理

公開文書を、製品を理解、利用、評価、再現するために必要な内容へ絞る。

1. `README.md`をリポジトリの入口、`SPEC.md`を現在仕様、
   `ROADMAP.md`を将来方針として役割を固定する。
2. 新しい文書を作る前に、同じ読者と目的を持つ既存文書への統合を検討する。
3. 重複するデプロイ、UI、モデル評価の説明を、それぞれ最小限の文書へまとめる。
4. 内部作業、引継ぎ、調査途中の記録は`_ai/`へ移す。
5. 廃止した案や完了済みの作業経緯は、必要な結論だけを現在文書へ残し、
   詳細はGit履歴で確認する。
6. 文書の統合・削除と同時に、リンクと公開文書テストを更新する。

各公開文書の読者と役割が重複せず、現在仕様、将来方針、内部作業を
別々に確認できる状態を完了条件とする。

## それ以降の候補

- 長い文を短いフレーズに分けて個別に練習する。
- 失敗したフレーズを履歴から再練習する。
- 成功回数、試行回数、最後の判定を保存する。
- 苦手なフレーズを次回の練習で優先表示する。
- 音声認識結果だけでは判断できない発音要素の評価方法を検討する。

これらは、LLMによる比較・採点とローカル版・公開版の処理統一が完了してから、
優先順位を決める。

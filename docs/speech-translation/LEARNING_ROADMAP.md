# 多言語発音練習モード ロードマップ

## 位置づけ

現状のユーザー画面を単なる音声翻訳アプリとして見せると、ChatGPTや既存翻訳アプリで十分ではないか、という比較になりやすい。

今後のデモ価値は、翻訳そのものではなく、利用者が「母国語で言いたいこと」を学習対象言語で言えるようになるまで練習できる固定ループに置く。

目標は、汎用チャットでは毎回プロンプトや会話の流れに依存しやすい処理を、スマホ上の1タップUIとして固定することである。

初期実装の学習対象言語は `ja-JP`、`zh-CN`、`en-US` とする。つまり、日本語学習だけでなく、日本人が中国語や英語を練習する使い方も同じ画面で扱う。

## 体験の核

1. 利用者が母国語で言いたいことを話す。
2. アプリが選択された学習対象言語へ変換し、模範音声を再生する。
3. 学習対象言語のテキストを表示する。日本語の場合は、可能ならひらがな表示も使う。
4. 利用者が模範音声を真似して学習対象言語で発音する。
5. アプリがASRで聞き取った学習対象言語の文を表示する。
6. 目標文とASR結果の近さを判定する。
7. うまく言えなければ何回も練習し、諦める場合は次に言いたいことを母国語で話す。

このモードでは、利用者にとっての主目的は「翻訳音声を聞く」ではなく、「言いたいことを学習対象言語で言えるようになる」ことにする。

## ChatGPTとの差別化

ChatGPTでも、事前に適切な指示を出せば近い練習はできる可能性がある。そのため、このアプリは「ChatGPTに絶対できないこと」ではなく、「ChatGPTでやると面倒、ブレる、続きにくいこと」を固定UIで解決する。

- 母国語入力、学習対象言語への変換、模範音声、復唱録音、判定、再挑戦を1つの画面フローに固定する。
- 利用者が毎回プロンプトを書かなくても、同じ基準で練習できる。
- 失敗したフレーズ、成功したフレーズ、再挑戦回数を履歴として残せる。
- 日本語学習では、漢字が読めない利用者向けに、ひらがな、ルビ、音声再生を前面に出す。
- デモでは、汎用AIチャットではなく、言いたいことを対象言語で話す練習機として説明する。

## 速度方針

発音練習ループでは、レスポンス速度を優先する。

- 初期実装ではSeed-VCを使わない。
- ジョーク、効果音、大阪弁、バリエーションも練習モードでは既定OFFにする。
- 模範音声は一度生成したら録音単位でキャッシュし、再挑戦時に再生成しない。
- 判定は初期実装ではLLMに依存せず、ASR結果と目標文の正規化テキスト比較で行う。
- LLMによる詳しい助言は、初期実装の必須要件にしない。入れる場合も、発音判定結果の後段で非同期または任意表示にする。

Seed-VCや将来のVibeVoice対応は、変換デモや高品質音声デモでは有効だが、発音練習の最短ループでは遅延要因になりやすい。発音練習モードでは、まずOpenAI ASR/翻訳/TTSの高速経路を優先する。

## UI方針

言語自動判定だけに頼ると、母国語で新しい内容を話しているのか、日本語で復唱しているのかが曖昧になりやすい。初期実装では、明示的な操作でモードを分ける。

- `いいたいことをはなす`: 母国語入力として扱い、選択した学習対象言語へ変換する。
- `まねしてはなす`: 直前の目標文を復唱する練習として扱う。
- `もういちど`: 同じ目標文を再練習する。
- `つぎへ`: 現在の練習を終え、次の母国語入力へ戻る。

ユーザー画面では、練習中の主表示を以下に絞る。

- 模範音声の再生ボタン。
- 目標文。日本語の場合は、可能ならひらがなまたはルビ付き表示。
- 録音ボタン。
- ASRで聞き取られた文。
- `OK`、`もうすこし`、`ちがうかも` などの短い判定。
- 目標文との差分が分かる簡単なハイライト。

練習画面は、トップページの録音ボタンと同じ視覚フィードバックを使う。録音中の音量に反応する波形/レベルメーター、処理中のアニメーション、押せる状態の分かりやすさはトップページと揃える。発音練習は高速に反復する用途なので、PC/スマホともにできるだけ1画面内へ収め、説明文を増やしすぎない。

画面上の固定文言は、学習者の母国語として自然な表記を使う。日本語UIでも、ひらがなのみではなく通常の日本語を使う。ただし初回ロード時点ではユーザーの母国語が未判定なので、日本語、中国語、英語の短い併記で案内する。

模範音声には再生速度UIを置く。初期実装ではTTSを再生成せず、ブラウザの `playbackRate` でYouTube相当の `0.25x` から `2.0x` までをスライダーで切り替える。選択した速度はブラウザ側へ保存する。比較再生の分割は、文末だけの `句点` と、カンマや読点も含む `句読点` を選べるようにする。

学習対象言語、中国語ピンイン表示、模範音声の再生速度はブラウザ側に保存し、毎回選び直さなくてもよいようにする。母国語入力のASR結果も画面に出し、利用者が「言いたいこと」が正しく聞き取られたかを確認できるようにする。

中国語では、お手本文にピンインを併記できるようにする。UIは中国語のお手本生成時にピンイン生成を要求し、既定ではピンインを表示する。

復唱結果は、お手本文の直下に同程度の大きさで表示し、目標文と違って聞こえた部分を赤字にする。抜け落ちた目標文側の語句も赤い欠落マーカーとして表示する。差分箇所はクリック可能にする。比較ボタンでは複数文を概算で分割し、「お手本1文目→復唱1文目、お手本2文目→復唱2文目」の順に再生する。将来的には、タイムスタンプ付きASRを使って不一致部分だけの音声比較へ進める。

練習用マイクは、お手本や判定表示によって画面下へ移動させない。録音を繰り返す操作の位置が変わると、反復練習のテンポが落ちるため、復唱用マイクは画面上部の録音カード位置に固定する。

## 段階的実装

### Phase 1: 最小発音練習ループ

目的は、翻訳デモから学習デモへ価値軸を移すことである。

- `/practice` に既存トップページとは別の発音練習画面を置く。既存の `へんなへんかんアプリ` は当面そのまま残す。
- 利用者が `ja-JP`、`zh-CN`、`en-US` から学習対象言語を選ぶ。
- `いいたいことをはなす` では母国語入力として扱い、OpenAI ASRの自動判定とOpenAI翻訳で学習対象言語へ変換する。
- 変換後、OpenAI TTSで模範音声を作り、専用ボタンで再生できるようにする。
- `まねしてはなす` ではSeed-VCを使わず、ユーザーの復唱音声を対象言語指定でASRする。
- 目標文とASR結果をUnicode正規化、句読点/空白除去、小文字化で比較する。日本語ではカタカナをひらがなへ寄せ、中国語では一般的な繁体字/簡体字の字形差を同一扱いにする。
- 練習用履歴は通常の音声変換履歴と混ぜず、`/practice/admin` と `/api/practice-history` で分けて確認する。
- 類似度を `OK`、`もうすこし`、`ちがうかも` の3段階程度で返す。
- `もういちど` と `つぎへ` を表示する。
- 同じ目標文の模範音声は再利用する。

初期判定は発音そのものの音響採点ではなく、ASRがどの程度目標文として聞き取れたかを見る。これは厳密な発音評価ではないが、最初の練習ループとしては実用的であり、日本語、中国語、英語で同じ実装を使いやすい。

### Phase 2: 練習しやすさの改善

- 間違いが多い部分を、ひらがな単位または文節単位で目立たせる。
- 長い文を短いチャンクに分けて練習できるようにする。
- 模範音声をゆっくり再生する選択肢を追加する。
- 直近の失敗フレーズを履歴から再練習できるようにする。
- 成功回数、試行回数、最後の判定を履歴に保存する。

### Phase 3: 発音フィードバックの高度化

- ASR結果だけでなく、可能なら音素、モーラ、アクセントに近い単位で評価する。
- 日本語らしく聞こえなかった箇所を、短い助言として表示する。
- LLMによる説明は、判定後の補助として使う。
- 同じ意味の自然な別表現も許容するか、目標文の完全復唱を重視するかをモードで分ける。

### Phase 4: 学習履歴と復習

- 言えたフレーズ、苦手フレーズ、試行回数をユーザー別に保存する。
- 苦手フレーズを次回デモや練習で優先表示する。
- 母国語の入力文、日本語目標文、ASR結果、判定、音声を紐づけて保存する。
- 保存先は現在の簡易履歴だけでなく、将来的にはR2/D1などの永続保存へ分ける。

## API案

初期実装では、既存の翻訳APIとは別に練習判定用APIを追加する。

```text
POST /api/practice/prompts
POST /api/practice/attempts
```

`POST /api/practice/prompts` 入力:

- `audio`: 利用者が母国語で話した音声。
- `target_language`: `ja-JP`、`zh-CN`、`en-US`。

出力:

- `transcript`: 母国語入力のASR結果。
- `target_text`: 学習対象言語へ変換した文。
- `display_text`: 画面表示用テキスト。日本語ではひらがな表示を含められる。
- `audio_base64`: 模範音声。
- `timings_ms`: ASR、翻訳、TTSの処理時間。

`POST /api/practice/attempts` 入力:

- `audio`: 利用者が学習対象言語で復唱した音声。
- `target_text`: 目標文。
- `target_language`: `ja-JP`、`zh-CN`、`en-US`。

出力:

- `recognized_text`: ASRで聞き取った文。
- `similarity`: 0.0から1.0の類似度。
- `grade`: `ok`、`almost`、`retry`。
- `diff`: 目標文とASR結果の差分表示用データ。
- `timings_ms`: ASR、正規化、比較の処理時間。

## 判定ロジック案

初期実装では、過度に複雑な発音採点を避ける。

1. 目標文とASR結果をUnicode正規化する。
2. 日本語ではカタカナをひらがなへ寄せる。
3. 句読点、空白、記号を取り除き、英語は小文字化する。
4. 文字単位の編集距離または近い文字列類似度で類似度を出す。
5. 短文では完全一致に寄せすぎず、部分一致も加点する。
6. しきい値の初期案:
   - `0.85` 以上: `OK`
   - `0.65` 以上: `もうすこし`
   - それ未満: `ちがうかも`

LLMでの採点は、説明文生成や意味が同じ別表現の許容が必要になってから追加する。

## 今後の実装課題

### 採点基準

現状の採点は、正規化後テキストを `SequenceMatcher` で比較する簡易判定である。これは初期デモには十分だが、発音練習の評価としては次の弱点がある。

- 語順や文字の近さに引っ張られ、意味の抜け落ちや語句置換が軽く見える場合がある。
- 中国語では、同じ発音の表記揺れ、簡体字/繁体字の揺れ、ピンイン上は近いが文字が違うケースを十分に扱えていない。
- 長文で一部だけ抜けた場合、全体スコアが高く出やすく、利用者に「どこを直すべきか」が伝わりにくい。
- `ok`、`almost`、`retry` のしきい値が文の長さや言語ごとに調整されていない。

次の段階では、文字列全体の類似度だけでなく、文・句・語の単位で欠落、置換、挿入を数える。重大な欠落や別語への置換がある場合は、全体類似度が高くても判定上限を下げる。中国語については、文字比較だけでなくピンイン列も比較対象にし、表記差と発音差を分けて扱う。将来の参考実装候補として、macOS版学習アプリ側の `ChineseScriptNormalizer`、`ChinesePinyinTransliterator`、`ScriptNormalizedChineseSegmenter`、HSK seed辞書の設計を確認する。

macOS版のピンイン生成はApple Foundationの `CFStringTransform` を使うSwift実装であり、Python APIやCloudflare Workerへコードとして直接共有する前提にはしない。共通化する場合は、辞書JSONなどのデータ資産、ゴールデンケース、正規化ルールを共有し、各ランタイムでは `pypinyin`、OpenCC、JavaScript向け同等ライブラリなどを使って同じ入出力を満たす形を優先する。

### 句読点分割と比較再生

現状の比較再生は、フロントエンドで正規表現により `sentence` または `punctuation` の単位に分割し、音声全体の長さを文字数比で割り当てている。これはタイムスタンプを持たない初期実装としての暫定策であり、実際の発話速度やポーズとはずれる。

次の段階では、分割単位の設定をUIで分かりやすく残しつつ、音声範囲は文字数比ではなくTTSまたはASRのタイムスタンプを優先する。タイムスタンプが取れない場合は、強制アライメントまたは無音検出を組み合わせる。比較表示では、抜け落ちた目標文側の区間を必ず赤字で見せ、認識結果だけに差分が出る状態にしない。

## デモ時の説明

デモでは、以下のように説明する。

> これは単なる翻訳アプリではなく、言いたいことを日本語で言えるようになるまで練習するアプリです。母国語で言いたいことを話すと、日本語の模範文と音声が出ます。それを真似して話すと、どれくらい近く聞き取れたかを判定して、何回でも練習できます。

日本語以外を学習対象にする場合は、以下のように説明する。

> これは単なる翻訳アプリではなく、言いたいことを選んだ言語で言えるようになるまで練習するアプリです。母国語で言いたいことを話すと、選んだ言語の模範文と音声が出ます。それを真似して話すと、どれくらい近く聞き取れたかを判定して、何回でも練習できます。

現時点の `へんなへんかんアプリ` の遊び要素は、発音練習モードとは分ける。発音練習モードでは速度と分かりやすさを優先し、声質変換やジョークはデモ演出用の別モードとして扱う。

## 未確定事項

- `まねしてはなす` をボタンで明示するか、録音内容の言語判定で自動分岐するか。
- ASR結果が意味としては近いが文面が違う場合に、正解扱いにするか。
- 短い文と長い文で類似度しきい値を変えるか。
- 発音判定履歴をどの粒度で保存するか。
- VibeVoiceを発音練習の模範音声にも使うか、演出用TTSとして別扱いにするか。

初期実装では、明示ボタン、完全復唱寄りの判定、Seed-VCなし、OpenAI API高速経路を優先する。

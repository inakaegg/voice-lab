# 実装フェーズ

## 目的

この文書は、音声翻訳Webアプリを一気通貫で進めるための実装フェーズを定義する。各フェーズは、仕様、実装、テスト、確認コマンドをセットで完了させる。

## 今後の実装方針

- 主要価値は「入力話者の声に寄せた別言語音声」を返すことに置く。翻訳読み上げだけではなく、声質類似度、出力速度、自然な発話品質を優先して評価する。
- 次の大きな確認はGPU環境での実測とする。まずASR、翻訳、TTS、声質変換を同じ入力で分けて測り、採用モデルと常駐worker化の必要性を決める。
- 公開MVPでは、一般ユーザー向けUIと管理者向けUIを分ける。一般ユーザー向けは録音、変換、再生を中心に簡潔にし、モデル選択、処理時間、provider設定は管理者向けに寄せる。
- 最初のGPU確認では、FastAPIのWeb UIとAPIをまとめてGPU環境へ載せ、一通り動作するかを先に確認する。その後、公開MVPでは静的UI/API gatewayとGPU推論APIを分離する。
- 最終的な公開構成では、静的UIと管理者gatewayはCloudflare側、ASR、翻訳、TTS、声質変換はRunPod等のGPU環境側で扱う。
- API契約はローカルFastAPIとGPU推論APIで揃える。frontendはAPI base URLを切り替えられるようにし、RunPod固有の認証情報をブラウザへ出さない。
- モバイル対応は、まずレスポンシブWebまたはWebViewで成立させる。ネイティブアプリ化は、録音、再生、認証、配布の要件が固まってから別フェーズで判断する。
- Redis、Fargate、Terraform、GitHub Actionsなどの追加基盤は、非同期job管理、永続的な結果管理、運用自動化などの必要が明確になった段階で導入する。

## フェーズ0: ローカルAPI土台

状態: 完了

目的:

- 重いモデルを使わず、音声翻訳パイプラインの入出力形を固定する。
- ASR、翻訳、TTS、テキスト加工を差し替え可能な境界に分ける。

範囲:

- `id-ID -> ja-JP` と `ja-JP -> zh-CN` のルート制限。
- fake providerを使った `POST /api/translate-speech`。
- `append_suffix` テキスト加工。
- 処理時間、プロバイダ名、警告を返すレスポンス形。

完了条件:

- パイプライン単体テストが通る。
- APIテストが通る。
- `/health` と `POST /api/translate-speech` をローカルで確認できる。

検証:

```sh
python3 -m pytest
PYTHONPATH=src python3 -m uvicorn mo_speech.api:app --host 127.0.0.1 --port 8000
curl -sS http://127.0.0.1:8000/health
```

## フェーズ1: ブラウザ操作UI

状態: 完了

目的:

- ブラウザから録音または音声アップロードを行い、API結果を確認できる最小UIを作る。
- fake providerのまま、UI状態とAPI連携を検証する。

範囲:

- 入力言語と出力言語の選択。
- 音声ファイルアップロード。
- 可能ならブラウザ録音。
- `append_suffix` の有効/無効、付加文字列、単位の指定。
- 処理中、成功、失敗、空状態の表示。
- 文字起こし、翻訳文、加工後テキスト、音声再生、処理時間の表示。

完了条件:

- APIテストが維持される。
- UIからfake providerのAPIを呼び出せる。
- 主要状態のUIスモーク確認ができる。
- デスクトップ幅とモバイル幅でフォームと結果表示が崩れない。

検証:

- `python3 -m pytest`
- ローカルサーバ起動後、ブラウザでアップロードから再生まで確認する。
- ローカルサーバ起動後、デスクトップ幅とモバイル幅で表示確認する。

## フェーズ2: ローカル実プロバイダ

状態: 完了

目的:

- fake providerを実プロバイダへ段階的に差し替え、ローカルで短い音声の縦切りを成立させる。

範囲:

- `MODEL_CACHE_DIR` の扱いを固定する。
- ASRプロバイダを追加する。
- 翻訳プロバイダを追加する。
- 初期TTSプロバイダを追加する。
- providerごとの依存関係、ロード失敗、未設定時のエラー表示を整理する。
- ローカル実プロバイダの起動方法とキャッシュ方針を [LOCAL_PROVIDERS.md](LOCAL_PROVIDERS.md) に残す。

完了条件:

- fake providerのテストが維持される。
- 実プロバイダの単体テストまたはスモークテストがある。
- 短いfixture音声で、文字起こし、翻訳、音声出力まで確認できる。
- モデル本体とキャッシュがリポジトリに入らない。

検証:

- `python3 -m pytest`
- fixture音声を使ったローカルスモーク確認。
- モデルキャッシュがgit管理対象に入っていないことの確認。

## フェーズ3: 声質クローン

状態: 完了

目的:

- 入力話者の声を出力音声へ反映する。これはMVP完成形の主要価値として扱う。

範囲:

- 参照音声の抽出または入力音声の流用。
- 声質クローンTTSの候補評価。
- 必要に応じたTTS + 声質変換の二段構成。
- voice profileを保存する場合の同意、削除、保存先の仕様化。
- `voice_mode` のAPI契約と未対応providerのエラーを固定する。
- 保存方針とモデル選択は [VOICE_CLONE.md](VOICE_CLONE.md) に残す。

完了条件:

- `voice_mode=clone` と `voice_mode=convert` のproviderが実装される。
- 入力音声と出力音声の関係がUI/APIで分かる。
- 短いサンプルで声質クローン経路のスモーク確認ができる。
- プライバシーと保存方針がdocsに反映される。

検証:

- `python3 -m pytest`
- 短い参照音声を使ったローカルスモーク確認。
- voice profile保存を行う場合は削除手順も確認する。

## フェーズ4: 応答速度改善

状態: 完了

目的:

- 「話し終えた直後に出力音声が話し始める」体験へ近づける。この評価はローカルCPUだけでは判断せず、GPU環境での実測を必須にする。

範囲:

- ASR、翻訳、TTS、声質変換の処理時間計測。
- ボトルネックの切り分け。
- 同期API、非同期job、ストリーミング出力の比較。
- モデルロード時間とウォーム状態の扱い。
- 計測方法と代表値は [LATENCY.md](LATENCY.md) に残す。

完了条件:

- Before/Afterの計測がdocsに残る。
- 主要処理のタイミングがレスポンスまたはログで確認できる。
- 改善対象と改善しない対象が明確になっている。

検証:

- 同じfixture音声で複数回計測する。
- cold startとwarm状態を分けて確認する。
- GPU環境でASR、翻訳、TTS、声質変換、totalを再計測する。

## フェーズ5: GPUデプロイ

状態: 一部完了 / 実デプロイ認証待ち

目的:

- ローカルで確認済みの構成をRunPodまたは代替GPUプラットフォームへ載せる。

範囲:

- Docker imageまたはプラットフォーム用エントリポイント。
- 初回確認では、FastAPIのWeb UIとAPIを同じGPU環境で動かす。
- モデル重みの保存先。
- 環境変数と秘密情報の扱い。
- ヘルスチェック。
- GitHub ActionsまたはCLI/APIによる更新手順。
- Dockerfile、RunPod handler、手順は [RUNPOD.md](../deployment/RUNPOD.md) に残す。

完了条件:

- ローカルとデプロイ先で同じAPI契約を保つ。
- モデル本体をリポジトリや大きいimage layerへ不用意に入れない。
- デプロイ手順と確認コマンドがdocsにある。

検証:

- デプロイ先の `/health`。
- デプロイ先Web UIから録音またはアップロードを使った一通りの動作確認。
- 短いfixture音声でのAPIスモーク確認。
- モデル保存先、起動時間、処理時間、費用見積もりの確認。

## フェーズ6: MVP仕上げ

- 主要ブラウザでの録音/再生確認。
- 既知の制限は [KNOWN_LIMITS.md](KNOWN_LIMITS.md) に残す。

完了条件:

- READMEからローカル起動と主要機能確認ができる。
- テストとスモーク確認が通る。
- 未対応範囲と既知の制限が公開docsに整理される。

検証:

- ローカルUIスモーク確認。
- デプロイ先APIスモーク確認。
